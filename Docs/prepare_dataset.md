# prepare_dataset.md

## Вкладка «Подготовка датасета»

Вкладка предназначена для создания готового датасета в формате YOLO из исходных изображений, аннотаций (файлы `.txt` в YOLO‑формате) или масок сегментации (PNG). Пользователь может задать преобразование размера изображений (с сохранением пропорций или растяжением), применить аугментации (флип, повороты, случайный поворот, сдвиг), настроить разбиение на train/val/test, перемапить ID классов и сгенерировать финальную структуру датасета с файлом `data.yaml`. Поддерживается три типа датасетов:
- **Метки (bbox, obb, seg)** – для детекции и сегментации с использованием YOLO‑аннотаций.
- **Маски (masks)** – для сегментации: исходные маски (PNG) конвертируются в полигоны YOLO‑seg.
- **Только изображения** – датасет без аннотаций (например, для тестирования или обучения без разметки).

### Основные возможности

1. **Загрузка исходных данных**
   - **Load Images** – выбор папки с изображениями (поддерживаются `.png`, `.jpg`, `.jpeg`, `.bmp`, `.tif`, `.tiff`, `.webp`).
   - **Load Labels** / **Load Masks** – в зависимости от выбранного типа датасета:
     - Для «Метки» – папка с `.txt`‑файлами (YOLO‑формат).
     - Для «Маски» – папка с PNG‑масками (одноканальные, значения пикселей – ID классов).
   - При загрузке автоматически сканируются пары «изображение – аннотация/маска» и отображаются в списке слева. Для масок проверяется валидность файлов, исключаются повреждённые.

2. **Настройка разбиения на Train / Validation / Test**
   - Три слайдера (Train, Validation, Test) в процентах, сумма всегда 100%. При изменении одного слайдера остальные корректируются автоматически.
   - Под слайдерами отображается фактическое количество снимков в каждом сплите (с учётом множителя аугментаций).
   - Общее количество снимков после аугментаций также показывается.

3. **Preprocessing (предобработка изображений)**
   - **Keep original size** – сохранять исходный размер (без ресайза).
   - **Resize to square (keep ratio, pad)** – привести к квадрату заданного размера (например, 640×640) с сохранением пропорций и добавлением чёрных/белых полей (фон настраивается: Black / White).
   - **Resize to square (stretch)** – растянуть изображение до квадрата без сохранения пропорций (аннотации остаются нормализованными, т.к. пропорции искажаются, но YOLO‑формат допускает это).
   - Поле ввода размера (32…4096) активно для двух последних режимов.

4. **Augmentation (аугментации)**
   - **Horizontal Flip** / **Vertical Flip** – случайное отражение по горизонтали/вертикали (вероятность 50%).
   - **Rotate 90/180/270°** – случайный поворот на 90, 180 или 270 градусов.
   - **Random Rotation (±)** – случайный поворот на угол до указанного максимума (по умолчанию 5°). Параметр задаётся в градусах.
   - **Random Shear (±)** – случайный сдвиг (shear) по X и Y до указанного угла (по умолчанию 5°).
   - **Random Crop** – временно отключён (функционал не реализован).
   - Все аугментации применяются как к изображению, так и к аннотациям/маскам с корректным пересчётом координат (для OBB и полигонов – преобразование всех точек, для detect – пересчёт bounding box).

5. **Augmentation multiplier (множитель датасета)**
   - Активен, если включена хотя бы одна аугментация.
   - Позволяет увеличить датасет в K раз (K от 2 до 10). Для каждой исходной пары генерируется K–1 дополнительных копий с различными случайными аугментациями (уникальность проверяется по хэшу изображения, чтобы избежать дубликатов).
   - При отсутствии аугментаций множитель неактивен (фиксирован 1x).

6. **Class Labels (маппинг классов)**
   - Таблица, отображающая:
     - **New ID** – новый ID класса в итоговом датасете (можно редактировать).
     - **Original ID** – исходный ID из аннотаций/масок (только для чтения).
     - **Name** – текстовое имя класса (можно редактировать).
     - **Include** – чекбокс: включать ли этот класс в датасет.
   - Для типа «Метки»: из всех аннотаций собираются уникальные class ID; по умолчанию все включены, New ID совпадает с Original ID. Пользователь может изменить New ID (например, перенумеровать классы) или исключить ненужные классы.
   - Для типа «Маски»: анализируются все маски, собираются уникальные значения пикселей (кроме 0 – фон). New ID генерируются автоматически последовательно от 0 (для фона не создаётся класс). Пользователь может исключить классы (снять Include), тогда соответствующие пиксели в масках будут игнорироваться.
   - Для типа «Только изображения» таблица классов отсутствует.

7. **Генерация датасета**
   - Кнопка **Generate Dataset** – запускает процесс в отдельном потоке (`DatasetGeneratorThread`).
   - Во время генерации отображается прогресс‑бар, кнопка «Generate» меняется на «Генерация...», появляется кнопка **Cancel** для прерывания.
   - Все операции выполняются в папке, выбранной через **Save YAML** (кнопка выбора выходной папки).
   - После успешной генерации в выходной папке создаётся структура:
     ```
     output_folder/
     ├── images/
     │   ├── train/
     │   ├── val/
     │   └── test/
     ├── labels/          (для типов «Метки» и «Маски»)
     │   ├── train/
     │   ├── val/
     │   └── test/
     ├── masks/           (только для типа «Маски» – исходные маски, опционально)
     ├── data.yaml        (файл конфигурации YOLO)
     ├── classes.txt      (список имён классов построчно)
     └── notes.json       (дополнительная информация о классах)
     ```
   - Для типа «Маски»: происходит конвертация масок в YOLO‑полигоны с помощью функции `convert_segment_masks_to_yolo_seg_manual`. Исходные маски остаются в папке `masks/` (опционально), а полигоны сохраняются в `labels/`.
   - В случае ошибки или отмены пользователем выводится соответствующее сообщение.

8. **Лог и управление**
   - **Показать лог** – текстовое окно с подробными сообщениями о каждом этапе (сканирование пар, применение аугментаций, конвертация, распределение).
   - Кнопка **Open Folder** – открывает выходную папку в проводнике (активна после выбора папки).
   - Прогресс‑бар отображает процент завершения генерации.

### Процесс генерации (подробно)

1. **Сканирование пар** – на основе выбранных папок формируется список `(img_path, label_path)` или `(img_path, mask_path)`.
2. **Маппинг классов** – на основе таблицы классов создаются словари `class_mapping` (original→new) и `mask_class_remap` (пиксельное значение→new).
3. **Генерация в временную папку `all/`** – для каждой пары:
   - Загружается изображение и (при наличии) аннотации/маска.
   - Применяются преобразования в порядке: сначала простые аугментации (флип, поворот 90°), затем сложные (случайный поворот, shear), затем ресайз.
   - Сохраняется оригинальная копия (без аугментаций) и, если `aug_multiplier > 1`, дополнительные копии с различными комбинациями аугментаций (уникальность контролируется хэшем изображения, чтобы не создавать идентичные файлы).
   - Для типа «Метки» аннотации пересчитываются согласно преобразованиям, сохраняются в `.txt`.
   - Для типа «Маски» маски ресайзятся и сохраняются во временную папку `masks/`.
4. **Конвертация масок в YOLO‑полигоны** (только для типа «Маски»):
   - Используется `convert_segment_masks_to_yolo_seg_manual`, которая для каждого файла маски находит контуры каждого класса, преобразует их в нормализованные полигоны и сохраняет в `.txt`.
   - Требуется установленный `ultralytics`.
5. **Разбиение на train/val/test** – все сгенерированные пары перемешиваются и распределяются согласно процентам. Файлы физически перемещаются из временной папки в соответствующие подпапки.
6. **Создание файлов конфигурации** – `data.yaml`, `classes.txt`, `notes.json`.
7. Удаление временной папки `all/`.

### Особенности реализации

- **Обработка аннотаций при аугментациях**: используется модуль `augmentation_base` и `augmentation_random`, поддерживающий типы `detect`, `obb`, `segment`. Для `obb` после аффинных преобразований применяется `repair_obb_to_rectangle`, чтобы гарантировать прямоугольную форму (иначе после сдвига/поворота OBB может перестать быть прямоугольником).
- **Уникальность аугментированных копий**: вычисляется MD5‑хэш результирующего изображения; если копия с таким же хэшем уже создана, попытка повторяется (до 10 раз). Это предотвращает дублирование при случайном совпадении параметров.
- **Конвертация масок**: требуется установка `ultralytics`. Функция `convert_segment_masks_to_yolo_seg_manual` написана вручную (не использует встроенную `convert_segment_masks_to_yolo_seg` из‑за особенностей формата). Она ожидает, что пиксели маски – целые числа, соответствующие исходным ID классов, и перемапливает их согласно `pixel_to_class`.
- **Игнорирование класса 0** в масках: фон не включается в список классов и не сохраняется в полигоны.
- **Аугментации для типа «Только изображения»** применяются аналогично, но без аннотаций.

### Интерфейс пользователя

- **Левая панель**: список найденных пар (с указанием типа для меток и масок).
- **Правая панель** (с прокруткой):
  - Блок Train/Val/Test Split
  - Блок Preprocessing
  - Блок Augmentation
  - Блок Augmentation multiplier
  - Блок Class Labels (таблица)
  - Кнопки «Generate Dataset», «Cancel», «Показать лог»
  - Кнопка «Open Folder» (рядом с полем пути вывода)
- **Нижняя часть**: лог и прогресс‑бар (скрыты по умолчанию, открываются кнопкой).

### Зависимости

- `ultralytics` – необходима для конвертации масок в YOLO‑полигоны. Если не установлена, соответствующая функция выдаст ошибку.
- OpenCV, numpy, PyQt5, yaml, shutil, random, hashlib – все стандартные для проекта.

### Примечания

- При выборе типа «Только изображения» кнопка «Load Labels» становится неактивной, таблица классов не отображается.
- Если в паре «изображение – маска» файл маски совпадает с файлом изображения (по имени и расширению), такая пара пропускается (предотвращение зацикливания).
- Для масок поддерживаются форматы PNG, JPG, JPEG, BMP, TIFF. Рекомендуется использовать PNG с lossless‑сжатием.
- Файл `notes.json` создаётся для совместимости с некоторыми инструментами визуализации, содержит категории и метаинформацию.
- Пути в `data.yaml` записываются в абсолютном формате (с заменой обратных слешей на прямые для совместимости).
